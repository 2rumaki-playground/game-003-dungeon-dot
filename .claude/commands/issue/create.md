# Issue作成

引数 `$ARGUMENTS` にIssueの概要（1行の要約）が渡される。

## 手順

### 1. コンテキスト収集

引数の概要をもとに、関連しそうなコードや仕様を調査する。

- `docs/spec/` 配下の仕様ドキュメントを確認
- `src/` 配下の関連コードを確認
- 既存Issueとの重複チェック: `gh issue list --state open --limit 30 --json number,title`

重複がある場合はユーザーに報告し、続行するか確認する。

### 2. Issue内容の確定

AskUserQuestion を使い、Issue内容を確認する。

##### ラベル（自動判定）

調査結果から適切なラベルを自動で選択する（複数該当する場合は複数付与可）。ユーザーには質問しない。

| ラベル | 用途 |
|--------|------|
| `bug` | バグ修正 |
| `enhancement` | 新機能・機能追加 |
| `documentation` | 仕様・ドキュメント |
| `refactor` | リファクタリング |
| `ui/ux` | UI/UXの改善・演出 |
| `perf` | パフォーマンス改善 |
| `test` | テストの改善 |

#### 詳細の確認

調査結果をもとに以下の草案を提示し、ユーザーに確認・修正してもらう：

- **ラベル**: 自動判定したラベル（変更可）
- **タイトル**: 何をするかを1行で（日本語、50文字以内）
- **背景/問題**: なぜこのIssueが必要か
- **該当箇所**（仕様の曖昧さに関するIssueの場合は **参照**）: 関連ファイル・仕様の参照（調査で見つかったもの）
- **受け入れ条件**: 何が達成されれば完了か（チェックリスト形式）

草案を提示したうえで「この内容で作成しますか？修正があれば教えてください」と確認する。

### 3. Issue作成

確定した内容で `gh issue create` を使って Issue を作成する。

- 通常のIssue: 草案確認で確定したセクション構成に従う
- 仕様の曖昧さに関するIssue: CLAUDE.md の「仕様の穴は可視化」セクションのフォーマットに従う

### 4. 結果報告

作成されたIssueのURLを表示する。

## ルール

- すべての出力は日本語
- タイトルは日本語、50文字以内
- 受け入れ条件には必ず「既存テストが全て通る」を含める（documentation ラベルのみの場合は除く）
- 仕様の曖昧さに関するIssueの場合、CLAUDE.md の「仕様の穴は可視化」セクションに従う
- 調査で関連する仕様ドキュメントが見つかった場合、受け入れ条件に仕様の更新も含める
