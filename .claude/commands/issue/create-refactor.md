# リファクタリング課題の調査・Issue起票

## コンテキスト

### 調査対象

- 引数: `$ARGUMENTS`
- 引数に調査対象のディレクトリやスコープが指定されていればそれを対象とする
- 引数が空の場合は `src/` 全体を対象とする

## 指示

コードベースのリファクタリング候補を網羅的に調査し、課題ごとに個別のGitHub Issueを起票してください。

### 1. コードベースの調査

Exploreエージェントを使い、以下の観点でリファクタリング候補を **網羅的に** 調査してください:

- **定数の重複**: 同じ値が複数ファイルに定義されているケース
- **コード重複**: 同じロジック・パターンが複数箇所にコピーされているケース
- **パターンの不統一**: 似た処理で異なる実装パターンが混在しているケース
- **型安全性の改善余地**: 型定義の重複、Any型の使用、より厳密な型にできる箇所
- **関心の分離不足**: 1つのファイル・関数が複数の責務を持っているケース
- **レイヤー違反**: domain層がpyxelやwandbに依存しているケース

各候補について、**具体的なファイルパス・行番号・重複内容**を記録すること。

### 2. 調査結果の整理・提示

発見した候補を以下の形式で一覧にまとめ、ユーザーに提示してください。

**依存関係を考慮した順序決定**:

Issue間の依存関係を分析し、**番号が若い順にPR作成すれば自然に依存が解決される順序**で並べること。具体的には:

1. 他のIssueの前提となるもの（共通モジュール導入、定数の一元化など）を先にする
2. 前提に依存するもの（一元化された定数を使う側のリファクタリングなど）を後にする
3. 依存関係がないもの同士は、影響範囲が広い順に並べる

依存関係がある場合はテーブルの「依存」列に明示する。

| 順序 | タイトル | 概要 | 該当箇所 | 依存 |
|------|---------|------|---------|------|
| 1 | ... | ... | ファイル名 x N箇所 | なし |
| 2 | ... | ... | ファイル名 x N箇所 | #1 |

ユーザーの確認を得てから次に進むこと。ユーザーが一部を除外・順序変更した場合はそれに従う。

### 3. Issue起票

承認された順序で、**依存関係の先頭から順に** Issueを起票してください。起票順 = Issue番号の昇順 = 作業順となるようにする。

```
gh issue create \
  --title "refactor: <日本語の簡潔な説明>" \
  --body "$(cat <<'EOF'
## 背景

<なぜこのリファクタリングが必要か。保守性・変更容易性の観点で説明>

## 該当箇所

<重複・問題のあるファイルと行番号の具体的なリスト>

## 受け入れ条件

- [ ] <完了条件1>
- [ ] <完了条件2>
- [ ] 既存テストが全て通る
EOF
)"
```

**Issue起票のルール**:
- 1つの課題 = 1つのIssue（粒度を分ける）
- タイトルは `refactor:` プレフィックス付き、日本語、50文字以内
- 受け入れ条件には必ず「既存テストが全て通る」を含める
- 該当箇所にはファイルパスと行番号を具体的に記載する
- Issueの本文にコード例を含める場合は変更前のパターンのみ記載し、実装方針は受け入れ条件に含めない（実装時の判断余地を残す）
- **依存関係がある場合**: bodyの末尾に `依存: #<先行Issue番号>` を記載する（例: `依存: #195`）。これにより、先行Issueのマージ後に着手すべきことが明確になる
- **起票順序は厳守**: 必ず依存先 → 依存元の順に1件ずつ起票し、Issue番号が若い順に着手すれば依存が自然に解決されるようにする

### 4. 結果報告

起票したIssueの一覧を作業順序とともに報告してください:

| 作業順 | Issue | タイトル | 依存 |
|--------|-------|---------|------|
| 1 | #xxx | refactor: ... | なし |
| 2 | #yyy | refactor: ... | #xxx |

「Issue番号の若い順に `/pr:create-from-issue` で対応すれば依存関係が自然に解決される」ことを確認してください。
